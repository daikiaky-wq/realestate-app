## Git 運用ルール

### 基本方針

- **コードを変更するたびに GitHub へプッシュする。**
- `main` ブランチへの直接プッシュは禁止。必ず feature ブランチを切って PR を作成する。
- コミットメッセージは日本語または英語で、変更内容が一目で分かる形で記述する。

### ブランチ命名規則

```
feature/<機能名>   # 新機能
fix/<バグ名>       # バグ修正
refactor/<対象>    # リファクタリング
docs/<対象>        # ドキュメント
```

### 変更のたびに行う手順

1. 変更後すぐにステージングしてコミットする

   ```
   git add <変更ファイル>
   git commit -m "変更内容の説明"
   ```

2. 現在のブランチを GitHub へプッシュする

   ```
   git push origin <current-branch>
   ```

3. 新しい機能・修正が完了したら GitHub 上で PR を作成し、レビューを経てマージする。

### コミットの粒度

- 一つの論理的な変更 = 一コミット。まとめすぎず、細かすぎず。
- WIP コミットは許容するが、PR マージ前にはスカッシュまたは整理する。

---

## 開発スタイル

- 不要なコメント・TODO・コンソールログをコードに残さない。
- 新しいファイルを追加する前に、既存ファイルへの編集で対応できないか検討する。
- テストは変更した機能に対してのみ書く。既存テストが壊れていないことを確認してからプッシュする。
